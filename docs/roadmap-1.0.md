# 多平台视频发布工具 1.0 路线图

这份路线图的目标，是把当前“已经验证可跑通真实发布流程的实验版”，推进成“可以稳定日常使用的 1.0 产品版”。

当前版本已经具备：

- 单视频草稿创建
- 标题、简介、封面候选生成
- 本地审核页确认
- 优先复用当前 Chrome 登录态
- 小红书、抖音、视频号真实发布链路已验证打通

但距离产品版，还差“稳定性、可观测性、可恢复性、可维护性”这四层能力。

## 1.0 版本目标

1.0 的定义建议是：

- 支持单视频稳定发布到小红书、抖音、视频号
- 发布前对主要平台限制进行明确校验
- 发布后能够记录每个平台的结果和失败原因
- 审核页足够支撑运营日常使用
- 失败任务可定位、可重试
- 文档和首次安装引导对非开发者也足够友好

## 优先级分层

### P0：必须完成

这些模块不完成，项目仍然更接近“实验版”而不是“产品版”。

1. 发布前校验层
2. 三平台发布成功判定
3. 错误日志和截图归档
4. 草稿列表与发布历史
5. 审核页增强
6. 失败重试机制

### P1：建议完成

这些模块会显著提升日常使用体验，但不一定阻塞 1.0 首次发布。

1. 更强的标题和文案建议
2. 封面候选与封面文案增强
3. 多账号与会话管理
4. 更明确的首次安装与授权向导

### P2：可延后

这些模块更偏 1.1 或后续版本。

1. 批量发布
2. 定时发布
3. 多人协作
4. 数据回收与发布后分析
5. 云端部署

## 阶段规划

## 阶段 1：稳定发布内核

目标：把“能发”变成“稳定发”。

### 需要完成

1. 完整发布前校验
   - 标题长度
   - 封面尺寸/比例
   - 视频格式、大小、时长
   - 平台必填字段
   - 发布时间与发布模式校验

2. 三平台成功判定标准化
   - 小红书：发布完成页或内容管理返回态
   - 抖音：作品管理页返回态
   - 视频号：作品列表页返回态

3. 错误归档机制
   - 错误类型
   - 错误文案
   - 页面截图
   - 草稿快照
   - 平台执行耗时

4. 失败重试入口
   - 单平台重试
   - 不影响其他已成功平台

### 建议修改的代码区域

- `src/mvpublisher/models/draft.py`
- `src/mvpublisher/approval/service.py`
- `src/mvpublisher/publishers/base.py`
- `src/mvpublisher/publishers/runner.py`
- `src/mvpublisher/publishers/xiaohongshu.py`
- `src/mvpublisher/publishers/douyin.py`
- `src/mvpublisher/publishers/wechat_channels.py`

## 阶段 2：补齐运营工作台

目标：让运营同学愿意每天用，而不是只靠命令行和临时页面。

### 需要完成

1. 草稿列表页
   - 草稿状态
   - 最近更新时间
   - 平台选择
   - 是否已批准

2. 发布历史页
   - 每个平台结果
   - 发布时间
   - 错误摘要
   - 快速重试入口

3. 审核页增强
   - 分平台字段区域
   - 标题候选一键选择
   - 封面候选与自定义封面更清晰
   - 发布前摘要卡片

### 建议修改的代码区域

- `src/mvpublisher/web/app.py`
- `src/mvpublisher/web/templates/index.html`
- `src/mvpublisher/web/templates/draft_detail.html`
- `src/mvpublisher/storage/drafts.py`

## 阶段 3：提升内容建议能力

目标：减少人工改文案和挑封面的时间。

### 需要完成

1. 更强的标题生成
   - 平台约束感知
   - 标题长度自适应
   - 多风格候选

2. 平台差异文案
   - 小红书版
   - 抖音版
   - 视频号版

3. 封面增强
   - 更好的关键帧挑选
   - 封面文案建议
   - 封面候选排序

### 建议修改的代码区域

- `src/mvpublisher/media/video_skill_adapter.py`
- `src/mvpublisher/media/cover_frames.py`
- `src/mvpublisher/suggestions/generator.py`

## 阶段 4：可持续维护能力

目标：降低后续接手、调试和扩展成本。

### 需要完成

1. 多账号与会话管理
   - 平台账号标识
   - 会话健康检查
   - 登录失效提示

2. 首次安装与授权向导
   - 系统权限提示
   - Chrome 登录态提示
   - 平台业务权限提示

3. 回归与诊断工具
   - 真机回归脚本
   - 关键页面截图校验
   - 常见失败原因排查指引

4. 文档同步机制
   - README
   - Skill
   - 路线图

### 建议修改的代码区域

- `src/mvpublisher/sessions/browser_reuse.py`
- `src/mvpublisher/sessions/playwright_fallback.py`
- `README.md`

## 推荐实施顺序

如果只按投入产出比来排，建议顺序是：

1. 发布前校验层
2. 三平台成功判定
3. 错误日志与截图归档
4. 审核页增强
5. 草稿列表与发布历史
6. 失败重试机制
7. 建议生成增强
8. 多账号与会话管理

## 里程碑建议

### M1：稳定版内核

完成条件：

- 三平台发布前校验可用
- 发布成功判定明确
- 失败日志可查
- 单平台可重试

### M2：可日常使用版

完成条件：

- 草稿列表页可用
- 审核页体验明显改善
- 发布历史和结果查看可用

### M3：1.0 正式版

完成条件：

- 内容建议增强完成首版
- 首次安装与授权引导可用
- 文档对非开发用户足够清晰
- 回归验证流程稳定

## 一句话建议

下一步不要先追求“更多平台”或“更自动”，而是先把“稳定发、可回看、可重试、好确认”这四件事做好。那才是从实验项目走向 1.0 产品版的关键。
